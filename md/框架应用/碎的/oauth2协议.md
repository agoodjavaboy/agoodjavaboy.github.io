# Oauth2

[oauth2的说明文档](https://datatracker.ietf.org/doc/html/rfc6749)

Oauth是一种第三方认证规范，在登录网站的时候，很多存在使用第三方账号登录的方式，点击相应的登录方式将采用第三方的认证页面来登录第三方网站，接着将第三方平台的信息携带到登录网站中完成登录。

这样的登录方式可以让用户使用同一平台的账户在很多的网站中进行登录，并且登录的网站中也无需保存用户的用户名和密码以及验证流程，而是交由第三方平台完成。

## 优点

Oauth2出现是为了解决第三方登录时所导致的问题：

1. 如果要第三方平台授权登录，通常要将用户的凭证保存到本服务中。
2. 就算把密码交给了本服务，则在服务中仍需要编写足够安全的认证逻辑。
3. 因为用户凭证已经完全获得，则第三方再无法控制本服务对第三方资源的获取。
4. 第三方不能收回本服务中任何已经授权的凭证，除非第三方更改密码。
5. 第三方授权的任何一个服务泄露用户凭证都将导致用户凭证完全泄露。

## 角色

Oauth2协议中规定了几种角色，来描述认证流程中的每个环节节点：

1. 资源所有者：具有对资源授权的实体，通常指用户。
2. 资源服务器：保存实体的资源所在的服务器，能够通过令牌来验证并返回资源。
3. 客户端：发出资源访问的程序，可以是服务器或者浏览器或者任何软件。
4. 授权服务器：在认证成功后向客户端发送令牌从而获取授权。

其中授权服务器和资源服务器可以是同一个服务器，或者时单独的服务器。

## 流程

Oauth2提供的访问规范如下：

```
     +--------+                               +---------------+
     |        |--(A)- Authorization Request ->|   Resource    |
     |        |                               |     Owner     |
     |        |<-(B)-- Authorization Grant ---|               |
     |        |                               +---------------+
     |        |
     |        |                               +---------------+
     |        |--(C)-- Authorization Grant -->| Authorization |
     | Client |                               |     Server    |
     |        |<-(D)----- Access Token -------|               |
     |        |                               +---------------+
     |        |
     |        |                               +---------------+
     |        |--(E)----- Access Token ------>|    Resource   |
     |        |                               |     Server    |
     |        |<-(F)--- Protected Resource ---|               |
     +--------+                               +---------------+
```

A. 客户端也就是浏览器向用户发出授权请求。

B. 客户端收到用户的授权之后，浏览器就将获得用户所收到的凭证。

C. 客户端访问授权服务器，传入授权凭证来获取token令牌。

D. 授权服务器在验证凭证有效的情况下颁发令牌给浏览器。

E. 客户端可以通过拿到的令牌访问受保护的资源。

F. 如果资源服务器验证令牌有效则返回资源内容。

实际上，第一步也就是在点击微信登录之后跳转到一个微信扫码登录的页面，扫码完成微信将返回一个认证的凭证给本网站，网站可以拿着凭证去微信的授权服务器拿到令牌，并且通过令牌拿到微信用户信息。在用户眼里只是扫码了一次，也就是A和B步，其他步骤由本服务器自动完成。

本服务在拿到微信认证后的资源（通常是用户名头像啥的用户信息）之后，就可以把用户作为本服务的业务用户存到本地用户表中，当然也可以更新用户的信息，但这些业务类型操作就已经超出Oauth2的范围了。

## 术语和令牌分类

常用的属于解释：

1. 凭证：客户端的编号以及密码，用于确定用户和用户所访问的本网站的ID。
2. 令牌：一串随机字符串，访问资源所用。
3. 作用域：用户获得令牌后，此令牌只能访问部分开放资源。

其中令牌是有时间限制的，所以要不断地更新令牌，更新的操作也将使用令牌，所以令牌也将被分为多个类：

1. 授权码：用来获取令牌。
2. 访问令牌：用于访问资源所用。
3. 刷新令牌：用于刷新访问令牌，获取新的可用的令牌。
4. BearerToken：静态令牌，任何客户端使用此令牌都可以访问资源。
5. Proof of Possession Token：验证客户端是否对令牌有使用权。

## 授权模式

### 授权码模式

授权码模式可以用来获取访问令牌和刷新令牌，并且针对需要认证的服务的保密性进行了优化，主要体现在每次请求所携带的重定向路径中。基于重定向的方式，进行认证的服务必须与资源拥有者进行交互，并且能够接收到授权服务器的响应。

```
     +----------+
     | Resource |
     |   Owner  |
     |          |
     +----------+
          ^
          |
         (B)
     +----|-----+          Client Identifier      +---------------+
     |         -+----(A)-- & Redirection URI ---->|               |
     |  User-   |                                 | Authorization |
     |  Agent  -+----(B)-- User authenticates --->|     Server    |
     |          |                                 |               |
     |         -+----(C)-- Authorization Code ---<|               |
     +-|----|---+                                 +---------------+
       |    |                                         ^      v
      (A)  (C)                                        |      |
       |    |                                         |      |
       ^    v                                         |      |
     +---------+                                      |      |
     |         |>---(D)-- Authorization Code ---------'      |
     |  Client |          & Redirection URI                  |
     |         |                                             |
     |         |<---(E)----- Access Token -------------------'
     +---------+       (w/ Optional Refresh Token)
```

A. 客户端（也就是进行登录的服务）通过用户代理器（也就是浏览器）向资源拥有者访问授权服务器，携带客户端标识、请求规范、本地状态和重定向的路径，授权服务器授权成功或失败后将返回此路径到浏览器。

B. 授权服务器验证用户代理器（浏览器）的授权请求，并确定是否同意后同意或拒绝访问。

C. 如果资源所有者进行了授权，授权服务器就将访问请求中携带的重定向路径访问需要进行登录的服务中的端口，路径中还会包含授权码和客户端发送的本地状态。

D. 客户端收到授权码后，从授权服务器的获取令牌的接口来获取令牌。通过获得到的路径发出请求后，授权服务器就将进行授权验证。

E. 授权服务器对客户端进行验证，验证授权码，并确保接收到的重定向路径与C中的路径相匹配后，将颁发访问令牌和刷新令牌。

### 隐式授权模式

隐式授权用户获取访问令牌，但不能获取刷新令牌。并且针对特定的访问路径进行了优化，可以通过Javascript脚本语言在浏览器中就可以实现。

同样的基于重定向，本地服务必须能发送和接受请求。

隐式授权方式中不包含客户端身份信息，依赖资源所有者和认证成功后重定向的路径。因为访问资源的令牌将被放到重定向路径中，所以可以公开给同意设备的其他应用程序。

```
     +----------+
     | Resource |
     |  Owner   |
     |          |
     +----------+
          ^
          |
         (B)
     +----|-----+          Client Identifier     +---------------+
     |         -+----(A)-- & Redirection URI --->|               |
     |  User-   |                                | Authorization |
     |  Agent  -|----(B)-- User authenticates -->|     Server    |
     |          |                                |               |
     |          |<---(C)--- Redirection URI ----<|               |
     |          |          with Access Token     +---------------+
     |          |            in Fragment
     |          |                                +---------------+
     |          |----(D)--- Redirection URI ---->|   Web-Hosted  |
     |          |          without Fragment      |     Client    |
     |          |                                |    Resource   |
     |     (F)  |<---(E)------- Script ---------<|               |
     |          |                                +---------------+
     +-|--------+
       |    |
      (A)  (G) Access Token
       |    |
       ^    v
     +---------+
     |         |
     |  Client |
     |         |
     +---------+
```

A. 需要认证的服务通过浏览器向授权服务器发出请求，携带服务标识、请求范围、本地状态和一个重定向的路径，授权服务器授权或拒绝后会将此路径返回浏览器。

B. 授权服务器对资源所有者进行身份验证，确定用户同意或拒绝本地服务的访问请求。
